Complete guide to migrating the YouTube Analyst from Cloud Run to Google Kubernetes Engine (GKE). Covers cluster setup, Kubernetes manifests, Workload Identity, GitHub Actions CI/CD, Terraform infrastructure-as-code, and monitoring.

Project: deve-487713 Region: us-central1 Backend: FastAPI + ADK Frontend: nginx

Overview

The YouTube Analyst currently runs on Cloud Run — a fully managed serverless platform. Moving to GKE gives you full Kubernetes control: custom networking, persistent workloads, multi-container pods, advanced autoscaling, and the ability to run GPU workloads in the future.

This guide walks you through every step from creating the cluster to updating your GitHub Actions pipelines, with no existing GKE knowledge required.

Cloud Run vs GKE

1 Enable APIs & IAM One-time

1.1 Enable required GCP APIs

Run these commands in your terminal with gcloud authenticated to your project:

gcloud config set project deve-487713

gcloud services enable \
  container.googleapis.com \

Note container.googleapis.com is the new one — this is the GKE API. All others were already enabled for Cloud Run.

1.2 Create a Deployer Service Account (for GitHub Actions)

This SA is used by GitHub Actions to push images and deploy to GKE — similar to what you already have for Cloud Run.

# Create the deployer SA
gcloud iam service-accounts create gke-deployer-sa \

gcloud projects add-iam-policy-binding deve-487713 \

2 Create GKE Cluster One-time

2.1 Create the cluster

We use GKE Standard (not Autopilot) so you have explicit control over node pools. The cluster is regional for high availability across 3 zones.

gcloud container clusters create youtube-analyst-cluster \
  --project=deve-487713 \
  --region=us-central1 \
  --num-nodes=1 \
  --machine-type=e2-standard-2 \
  --disk-size=50 \
  --disk-type=pd-standard \
  --enable-autoscaling \
  --min-nodes=1 \
  --max-nodes=3 \
  --enable-autorepair \
  --enable-autoupgrade \
  --workload-pool=deve-487713.svc.id.goog \
  --logging=SYSTEM,WORKLOAD \
  --monitoring=SYSTEM \
  --release-channel=regularbash
Cost Warning This creates 1 node per zone = 3 nodes total (regional cluster). Each e2-standard-2 costs ~$49/mo = ~$147/mo total. For dev, use --num-nodes=1 --min-nodes=0 --max-nodes=2 in a single zone with --zone=us-central1-a instead of --region to reduce cost.

2.2 Get credentials (connect kubectl)

gcloud container clusters get-credentials youtube-analyst-cluster \
  --region=us-central1 \
  --project=deve-487713

# Verify connection
kubectl get nodesbash

2.3 Create a Kubernetes namespace

Keep the app isolated from system workloads:

3 Workload Identity One-time

Workload Identity lets your GKE pods authenticate to GCP APIs (Firestore, Vertex AI, etc.) without any SA key files. A Kubernetes Service Account is linked to a GCP Service Account — pods get credentials automatically.

Pod in GKE → uses Kubernetes SA "youtube-analyst-ksa" → linked to GCP SA "youtube-analyst-sa@deve-487713.iam.gserviceaccount.com" → GCP SA has roles: datastore.user, aiplatform.user, secretmanager.secretAccessor → No JSON key file needed anywhere

3.1 Create the GCP runtime Service Account

gcloud iam service-accounts create youtube-analyst-sa \
  --display-name="YouTube Analyst Runtime SA" \
  --project=deve-487713

# Grant the roles the app needs at runtime
for ROLE in \
  roles/datastore.user \
  roles/aiplatform.user \
  roles/secretmanager.secretAccessor \
  roles/logging.logWriter \
  roles/monitoring.metricWriter \
  roles/storage.objectAdmin; do
  gcloud projects add-iam-policy-binding deve-487713 \
    --member="serviceAccount:youtube-analyst-sa@deve-487713.iam.gserviceaccount.com" \
    --role="$ROLE"
donebash

3.2 Create the Kubernetes Service Account

kubectl create serviceaccount youtube-analyst-ksa \
  --namespace youtube-analystbash

3.3 Bind them together (the Workload Identity link)

# Allow the KSA to impersonate the GCP SA
gcloud iam service-accounts add-iam-policy-binding \
  youtube-analyst-sa@deve-487713.iam.gserviceaccount.com \
  --role=roles/iam.workloadIdentityUser \
  --member="serviceAccount:deve-487713.svc.id.goog[youtube-analyst/youtube-analyst-ksa]"

# Annotate the KSA so GKE knows which GCP SA to use
kubectl annotate serviceaccount youtube-analyst-ksa \
  --namespace youtube-analyst \
  iam.gke.io/gcp-service-account=youtube-analyst-sa@deve-487713.iam.gserviceaccount.combash

4 Kubernetes Manifests

Create the following directory structure in your repo:

k8s/ ├── namespace.yaml ← already created via kubectl, but good to have in code ├── backend-deployment.yaml ← FastAPI + ADK app ├── backend-service.yaml ← internal ClusterIP service ├── frontend-deployment.yaml← nginx static frontend ├── frontend-service.yaml ← internal ClusterIP service ├── ingress.yaml ← routes external traffic to both services └── configmap.yaml ← non-secret environment config

4.1 Backend Deployment

# k8s/backend-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: youtube-analyst-backend
  namespace: youtube-analyst
  labels:
    app: youtube-analyst-backend
spec:
  replicas: 2                        # 2 pods for high availability
  selector:
    matchLabels:
      app: youtube-analyst-backend
  template:
    metadata:
      labels:
        app: youtube-analyst-backend
    spec:
      serviceAccountName: youtube-analyst-ksa   # Workload Identity
      containers:
        - name: backend
          image: us-central1-docker.pkg.dev/deve-487713/youtube-analyst/app:latest
          ports:
            - containerPort: 8080
          envFrom:
            - configMapRef:
                name: youtube-analyst-config    # non-secret env vars
          env:
            - name: YOUTUBE_API_KEY
              valueFrom:
                secretKeyRef:
                  name: youtube-analyst-secrets
                  key: YOUTUBE_API_KEY
          resources:
            requests:
              cpu: "500m"          # 0.5 vCPU guaranteed
              memory: "512Mi"
            limits:
              cpu: "2000m"         # max 2 vCPU
              memory: "2Gi"
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: youtube-analyst-backendk8s/backend-deployment.yaml

4.3 Ingress & Load Balancer

The Ingress routes external traffic: /chat and /health go to the backend; everything else goes to the frontend.

# k8s/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: youtube-analyst-ingress
  namespace: youtube-analyst
  annotations:
    kubernetes.io/ingress.class: "gce"           # GKE's built-in L7 load balancer
    kubernetes.io/ingress.global-static-ip-name: "youtube-analyst-ip"
    networking.gke.io/managed-certificates: "youtube-analyst-cert"
spec:
  rules:
    - host: your-domain.com                      # replace with your domain
      http:
        paths:
          - path: /chat
            pathType: Prefix
            backend:
              service:
                name: youtube-analyst-backend
                port:
                  number: 80
          - path: /health
            pathType: Prefix
            backend:
              service:
                name: youtube-analyst-backend
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: youtube-analyst-frontend
                port:
                  number: 80k8s/ingress.yaml
No custom domain yet? Skip the host: field and the managed certificate annotation. GKE will provision an external IP and you can access the app via http://<EXTERNAL_IP>. Add the domain later.

5 Secrets & Config One-time

The YouTube API key is sensitive — store it as a Kubernetes Secret pulled from GCP Secret Manager.

5.1 Store the API key in GCP Secret Manager

echo -n "your-youtube-api-key" | gcloud secrets create YOUTUBE_API_KEY \
  --data-file=- \
  --project=deve-487713bash

5.2 Create Kubernetes Secret from Secret Manager

Pull the value from Secret Manager and create a K8s Secret:

YOUTUBE_KEY=$(gcloud secrets versions access latest \
  --secret=YOUTUBE_API_KEY \
  --project=deve-487713)

kubectl create secret generic youtube-analyst-secrets \
  --namespace=youtube-analyst \
  --from-literal=YOUTUBE_API_KEY="$YOUTUBE_KEY"

# Unset the variable immediately — don't leave it in shell history
unset YOUTUBE_KEYbash
Best Practice For production, use the Secrets Store CSI Driver with GCP provider to sync Secret Manager values directly into pods automatically — no manual kubectl create secret needed.

6 GitHub Actions CI/CD Updated Pipelines

Replace the current deploy.yml (which uses gcloud run deploy) with a GKE-aware pipeline.

6.1 New deploy.yml for GKE Backend

# .github/workflows/deploy.yml
name: Build and Deploy Backend to GKE

on:
  push:
    branches: [main]
    paths:
      - 'youtube_analyst/**'
      - 'Dockerfile'
      - 'pyproject.toml'
      - 'uv.lock'
      - 'k8s/backend-deployment.yaml'
  workflow_dispatch:

env:
  PROJECT_ID:   deve-487713
  REGION:       us-central1
  CLUSTER:      youtube-analyst-cluster
  NAMESPACE:    youtube-analyst
  IMAGE:        us-central1-docker.pkg.dev/deve-487713/youtube-analyst/app

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Authenticate to GCP
        uses: google-github-actions/auth@v2
        with:
          credentials_json: ${{ secrets.GKE_SA_KEY }}

      - name: Set up gcloud
        uses: google-github-actions/setup-gcloud@v2
        with:
          project_id: ${{ env.PROJECT_ID }}
          install_components: gke-gcloud-auth-plugin

      - name: Configure Docker
        run: gcloud auth configure-docker us-central1-docker.pkg.dev --quiet

      - name: Build Docker image
        run: |
          docker build \
            --tag ${{ env.IMAGE }}:${{ github.sha }} \
            --tag ${{ env.IMAGE }}:latest \
            .

      - name: Push Docker image
        run: |
          docker push ${{ env.IMAGE }}:${{ github.sha }}
          docker push ${{ env.IMAGE }}:latest

      - name: Get GKE credentials
        run: |
          gcloud container clusters get-credentials ${{ env.CLUSTER }} \
            --region ${{ env.REGION }} \
            --project ${{ env.PROJECT_ID }}

      - name: Update image in deployment
        run: |
          kubectl set image deployment/youtube-analyst-backend \
            backend=${{ env.IMAGE }}:${{ github.sha }} \
            --namespace ${{ env.NAMESPACE }}

      - name: Wait for rollout
        run: |
          kubectl rollout status deployment/youtube-analyst-backend \
            --namespace ${{ env.NAMESPACE }} \
            --timeout=300s.github/workflows/deploy.yml (updated)

7 Terraform for GKE Infrastructure as Code

Add GKE resources to your existing terraform/ setup to version-control the cluster itself.

7.1 Add to terraform/modules/youtube-analyst/main.tf

# ── GKE Cluster ──────────────────────────────────────────────────────────────
resource "google_container_cluster" "primary" {
  project  = var.project_id
  name     = "youtube-analyst-cluster"
  location = var.region            # regional = HA across 3 zones

  # Remove the default node pool — we manage our own
  remove_default_node_pool = true
  initial_node_count       = 1

  workload_identity_config {
    workload_pool = "${var.project_id}.svc.id.goog"
  }

  logging_config {
    enable_components = ["SYSTEM_COMPONENTS", "WORKLOADS"]
  }

  monitoring_config {
    enable_components = ["SYSTEM_COMPONENTS"]
  }

  depends_on = [google_project_service.apis]
}

# ── Node Pool ─────────────────────────────────────────────────────────────────
resource "google_container_node_pool" "primary_nodes" {
  project    = var.project_id
  cluster    = google_container_cluster.primary.name
  location   = var.region
  name       = "primary-pool"
  node_count = 1

  autoscaling {
    min_node_count = 1
    max_node_count = 3
  }

  management {
    auto_repair  = true
    auto_upgrade = true
  }

  node_config {
    machine_type = "e2-standard-2"
    disk_size_gb = 50
    disk_type    = "pd-standard"

    service_account = google_service_account.cloud_run_sa.email
    oauth_scopes = [
      "https://www.googleapis.com/auth/cloud-platform"
    ]

    workload_metadata_config {
      mode = "GKE_METADATA"    # required for Workload Identity
    }
  }
}

8 Monitoring & Logging

GKE automatically sends logs and metrics to Google Cloud Monitoring (enabled via --logging and --monitoring flags during cluster creation).

Useful kubectl commands

# See all pods and their status
kubectl get pods -n youtube-analyst

# Stream live logs from backend pods
kubectl logs -f -l app=youtube-analyst-backend -n youtube-analyst

# Describe a pod (useful for crash debugging)
kubectl describe pod <pod-name> -n youtube-analyst

# Check resource usage
kubectl top pods -n youtube-analyst

# See rollout history
kubectl rollout history deployment/youtube-analyst-backend -n youtube-analyst

# Roll back to previous version
kubectl rollout undo deployment/youtube-analyst-backend -n youtube-analyst

# Get the external IP of the Ingress
kubectl get ingress -n youtube-analystbash

Horizontal Pod Autoscaler (optional)

Automatically scale pods based on CPU usage:

kubectl autoscale deployment youtube-analyst-backend \
  --namespace=youtube-analyst \
  --cpu-percent=70 \
  --min=2 \
  --max=10bash

Final Checklist

  • Enable container.googleapis.com and other required APIs
  • Create deployer SA (gke-deployer-sa) and add key to GitHub Secrets as GKE_SA_KEY
  • Create GKE cluster with Workload Identity enabled (--workload-pool)
  • Run kubectl create namespace youtube-analyst
  • Create runtime SA (youtube-analyst-sa) and bind to Kubernetes SA via Workload Identity
  • Store YOUTUBE_API_KEY in GCP Secret Manager and create K8s Secret
  • Create k8s/ folder with all manifests and run kubectl apply -f k8s/
  • Update deploy.yml and deploy-frontend.yml to use kubectl set image instead of gcloud run deploy
  • Add GKE cluster Terraform resource to terraform/modules/youtube-analyst/main.tf
  • Update frontend/index.html BACKEND URL to the GKE Ingress external IP or domain
  • Verify with kubectl get pods -n youtube-analyst — all pods should be Running
  • (Optional) Delete the old Cloud Run services to stop incurring charges
You're done! Once the Ingress external IP is assigned (kubectl get ingress -n youtube-analyst), your YouTube Analyst is running on GKE with zero-downtime rolling deployments, Workload Identity authentication, and horizontal autoscaling.
